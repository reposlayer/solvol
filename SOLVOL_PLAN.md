## Decisions Made

- Keep the existing Next.js/npm stack because it is already wired with terminal UI, React Query, and Polymarket data hooks.
- Keep the product read-only: no trade execution, order placement, custody, deposits, or withdrawals.
- Add a formal `MarketSource` abstraction and normalized terminal types instead of hardcoding Polymarket into new intelligence code.
- Prefer public Polymarket data where available and use deterministic seeded mock data whenever public APIs fail or external data is unavailable.
- Use Node's built-in test runner with `--experimental-strip-types` for focused TypeScript utility tests, matching the existing repo plan.
- Use the existing dense dark terminal design system instead of generating a fresh visual concept; the task is now implementation and product completion inside an existing UI direction.
- Keep fallback payloads explicit with `dataMode: "mock"` and `fallbackReason` so demo data is not mistaken for live claims.
- Keep a dedicated public Polymarket allowlist because the source of truth for Solvol Terminal is normalized public data, not credentials or LLM narration.
- Keep optional external source enrichment failure-tolerant; source timeouts should lower evidence density, not break the read-only market intelligence route.
- Keep Playwright artifacts local and ignored because dev-server file watching can otherwise create false browser-console failures.
- Keep the editorial masthead and ticker as the only global status surfaces; detailed counts and provider state belong in the relevant workspace panels to avoid duplicated information.
- Keep primary market actions in one article-header action row. Supporting rails should explain status and next steps, not duplicate buttons.
- Keep the redesigned terminal market-first: the first viewport should show the all-markets table and filters, while editorial panels and analysis modules support the browse flow instead of hiding it.
